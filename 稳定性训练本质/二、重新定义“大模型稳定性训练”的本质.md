本章承接全文的主线：**监控 → 发现预兆 → 定位根因 → 修复根因 → 根因未知或短期难修时使用 norm / clip / clamp / rescale 止血**。这里需要进一步明确：大模型稳定性训练不是单纯“防止数值变大”，而是对训练轨迹中的关键状态变量进行闭环控制，使模型始终处于**可学习、可解释、可复现、可恢复**的动态范围内。

---

### 2.1 工程化定义
**大模型稳定性训练**，是指在给定数据分布、模型结构、优化器、学习率调度、数值精度和分布式训练系统的条件下，对 LLM 训练过程中的 **loss、logits、attention scores、hidden states / residual stream、activation、gradient、parameter / weight、optimizer state、optimizer update、learning rate schedule、batch data distribution、sequence packing / mask、precision / loss scaling、distributed consistency、MoE routing** 等对象进行持续观测、异常识别、因果归因和干预，使训练过程满足以下性质：

1. **数值有限性**：关键张量不出现 NaN / Inf，不发生不可恢复的 overflow / underflow。
2. **尺度可控性**：activation、gradient、optimizer update、logits、attention scores 等对象的分布、范数、极值和 outlier ratio 不持续脱离合理区间。
3. **语义正确性**：数据、label、mask、position id、sequence packing、loss reduction、tokenizer / vocab 等训练语义正确。
4. **优化一致性**：learning rate、warmup、optimizer state、parameter group、gradient accumulation、update-to-weight ratio 等共同决定的实际更新量符合预期。
5. **系统一致性**：多卡、多机、FSDP / ZeRO、MoE expert parallel 等分布式状态在 rank 间保持一致。
6. **异常可定位性**：当出现 loss spike、gradient explosion、activation explosion、logits explosion、attention entropy collapse、NaN / Inf 或 expert collapse 时，能够定位其时序、层级、模块、batch、rank 或 expert 来源。
7. **干预最小性**：优先修复根因；只有在根因未知、短期无法完全修复或工程上需要保险机制时，才引入最小扰动的 norm / clip / clamp / rescale。

更简洁地说：

大模型稳定性训练的核心，不是让所有数值都“小”，而是让训练过程中的 **signal propagation、gradient flow、optimizer update、data semantics、precision range、distributed state** 始终保持在可学习且可恢复的工作区间内。

---

### 2.2 稳定性不是单点指标，而是训练轨迹属性
稳定性不能仅由某一个 step 的 loss 或 global grad norm 判断。一个训练过程是否稳定，取决于多个状态变量在时间维度上的联合演化：

[  
\theta_{t+1} = \theta_t + \Delta \theta_t  
]

其中：

[  
\Delta \theta_t = -\eta_t \cdot \mathcal{U}(g_t, s_t)  
]

+ (\theta_t)：当前参数；
+ (\eta_t)：当前 learning rate；
+ (g_t)：当前梯度；
+ (s_t)：optimizer state，例如 Adam 的 (m_t, v_t)；
+ (\mathcal{U})：优化器对梯度和状态的变换；
+ (\Delta \theta_t)：实际参数更新。

因此，仅监控 gradient norm 不够。真正影响训练轨迹的是：

[  
|\Delta \theta_t|,\quad \frac{|\Delta \theta_t|}{|\theta_t|},\quad \text{update direction},\quad \text{layer-wise update distribution}  
]

一个 batch 的 gradient norm 偏大，未必表示训练已经不稳定；但如果该 batch 同时导致以下现象，就更接近稳定性问题：

| 信号 | 判断含义 |
| --- | --- |
| update-to-weight ratio 明显偏离历史区间 | 实际参数更新可能过大 |
| Adam (v_t) 异常偏小 | adaptive update 可能被放大 |
| logits max / attention score p99 同步升高 | forward signal 可能已失控 |
| residual stream RMS 持续随层数或 step 增长 | signal propagation 可能异常 |
| rank-wise loss / grad 出现分歧 | 分布式一致性可能异常 |
| 同一 batch replay 稳定复现 spike | 数据、mask、precision 或代码语义可能有根因 |
| clip rate 持续升高 | clip 可能正在替代根因修复 |

### 2.3 稳定性对象的分层视角
大模型稳定性训练可以分成四层对象。不同对象对应不同问题来源，也对应不同处理优先级。

| 层级 | 关注对象 | 稳定性含义 | 典型异常 | 优先判断问题 |
| --- | --- | --- | --- | --- |
| 训练语义层 | data、label、loss mask、sequence packing、attention mask、position id、tokenizer / vocab | 训练目标和输入语义正确 | 某些 batch 稳定触发 loss spike；padding 被训练；packed sequence 串样本；label shift 错误 | 这是语义 bug，不能靠 clip 解决 |
| 信号传播层 | residual stream、hidden states、attention scores、activation、logits | forward signal 在深度方向和时间方向上可控 | residual RMS 增长、attention entropy collapse、logits 极端、activation outlier | 是架构尺度、初始化、attention、MLP 还是 mask 问题 |
| 优化动力学层 | gradient、optimizer state、optimizer update、learning rate、warmup、schedule、weight decay | backward signal 和实际 update 不失控 | grad norm 爆炸、Adam update 过大、resume 后发散、update-to-weight ratio 异常 | 是 LR、optimizer state、param group 还是 batch scale 问题 |
| 数值与系统层 | precision、loss scaling、overflow、kernel、distributed rank、FSDP / ZeRO、MoE routing | 数值实现和分布式状态一致 | fp16 overflow、rank divergence、checkpoint state mismatch、expert collapse | 是 precision、kernel、通信、shard state 还是 routing 问题 |


这个分层的工程意义是：**先排查语义正确性，再排查优化和信号尺度，最后再判断是否需要数值约束。**  
例如，padding token 未被 ignore 导致 loss spike，这属于训练语义层问题；使用 global grad clipping 只能降低梯度幅度，不能修复模型正在学习错误目标这一根因。

### 2.4 关键监控对象与稳定性含义
稳定性监控不应只记录平均值。至少需要记录 mean、std、min、max、p95、p99、norm、histogram、outlier ratio、non-finite ratio，并区分 global、layer-wise、head-wise、expert-wise、rank-wise 和 batch-wise 维度。

| 监控对象 | 在稳定性中的作用 | 推荐统计量 | 典型异常信号 | 第一判断问题 |
| --- | --- | --- | --- | --- |
| loss | 训练目标的直接反馈 | mean、p95、p99、per-token loss、per-sequence loss、spike ratio | loss spike、持续上升、NaN | 是数据 batch、label、mask、LR、precision 还是 optimizer 问题 |
| logits | 输出分布尺度 | max、RMS、entropy、top-1 probability、vocab outlier | logits 极端、entropy collapse、softmax 饱和 | 是 hidden RMS、lm_head、vocab/tokenizer 还是 CE 数值问题 |
| attention scores | attention softmax 前的关键尺度 | p95、p99、max、entropy、max attention probability | QK score 过大、attention one-hot、非法 attention | 是 Q/K norm、RoPE、mask、temperature 还是长序列问题 |
| hidden states / residual stream | 深层信号传播主路径 | layer-wise RMS、p99、max、depth trend | residual RMS 随层数或 step 增长 | 是 residual scaling、Pre-LN/Post-LN、初始化还是 LR 问题 |
| activation | MLP / attention 内部非线性信号 | RMS、max、kurtosis、outlier ratio、non-finite ratio | MLP gate outlier、activation explosion | 是 activation function、MLP init、residual branch 还是 precision 问题 |
| gradient | backward signal | global norm、layer norm、head/expert norm、non-finite ratio | grad norm 快速变大、某层梯度异常 | 是 loss spike 的结果，还是 backward 本身失控 |
| parameter / weight | 模型状态本体 | norm、RMS、max、row norm、spectral estimate | embedding/lm_head norm 漂移、某层权重异常 | 是 LR group、weight decay、optimizer state 还是 initialization 问题 |
| optimizer state | Adam (m, v)、step、state finite ratio | (m) norm、(v) min/max、state step、finite ratio | (v) 异常偏小、state NaN、resume 后 step 错 | 是 optimizer state 损坏、epsilon、beta 还是 checkpoint 问题 |
| optimizer update | 实际参数更新 | update norm、update-to-weight ratio、update-gradient cosine | grad 正常但 update 大；某层 U/W 过高 | 是 Adam transform、LR、param group 还是 state mismatch |
| learning rate / warmup / schedule | update 尺度控制器 | current LR、param-group LR、scheduler step、warmup progress | resume 后 LR 错；warmup 结束后 spike | 是 schedule 配置、global step、batch size scale 问题 |
| batch data distribution | 输入分布和目标分布 | length、source、language、sample weight、special token ratio | 某数据源或某长度段触发 spike | 是脏数据、异常权重、分布突变还是 sampler 问题 |
| sequence length / packing / mask | attention 和 loss 语义 | position id range、mask density、packed boundary、ignore ratio | 跨样本 attention、padding 被训练、长序列炸 | 是 mask、packing、position id 还是 RoPE scaling 问题 |
| precision / loss scaling / overflow | 数值范围 | loss scale、overflow count、non-finite tensor、dtype map | fp16 overflow、loss scale 频繁下降、kernel NaN | 是 dtype、loss scaling、softmax/exp/log 数值问题 |
| distributed consistency | 多卡状态一致 | rank-wise loss、grad norm、param checksum、state checksum | 单卡稳定多卡炸；某 rank 异常 | 是 all-reduce、FSDP/ZeRO、accumulation、rank data 问题 |
| MoE routing | expert 使用和路由稳定性 | expert load、router logits、router entropy、drop rate、aux/z-loss | expert collapse、router logits 爆、负载极不均 | 是 router、capacity、aux loss、expert state 还是数据分布问题 |

### 2.5 “正常范围”应如何定义
稳定性监控中的“正常范围”不应仅依赖固定阈值。不同模型规模、初始化、optimizer、batch size、sequence length、precision 和 architecture recipe 会改变指标尺度。更可靠的做法是为每个训练任务建立 **run-specific baseline**。

推荐使用五类基线。

| 基线类型 | 定义方式 | 适用对象 | 异常判断方式 |
| --- | --- | --- | --- |
| 初始化基线 | step 0 或前若干 step 的 forward/backward 统计 | activation、residual RMS、attention score、grad norm | 第一轮 forward/backward 就异常，通常指向初始化、mask、precision 或代码 bug |
| 稳定窗口基线 | warmup 中后段或 loss 平稳区间的 rolling statistics | loss、grad norm、update norm、logits、RMS | 当前值相对 rolling p95/p99 明显偏离 |
| 层间相对基线 | 相邻层、同类层之间的相对尺度 | residual、activation、grad、update | 某层相对邻层阶跃放大 |
| rank 间一致性基线 | 多 rank 同步前后的统计分布 | loss、grad、overflow、batch shape | 某 rank 与其他 rank 分布明显不同 |
| recipe 约束基线 | 由训练 recipe 给出的预期范围 | LR、warmup、tokens/update、loss scale、clip threshold | 当前配置与预期 recipe 不一致 |


工程上可以把异常分成三档：

| 等级 | 判断标准 | 处理方式 |
| --- | --- | --- |
| 黄色告警 | 指标偏离 rolling baseline，但未出现 NaN / Inf，也未影响 loss trend | 增加日志粒度，记录 batch、layer、rank、optimizer state |
| 橙色告警 | 指标持续偏离，或多个相关指标同步异常，例如 residual RMS、logits max、grad norm 同时升高 | 从最近 checkpoint replay，做 layer-wise / batch-wise 定位 |
| 红色告警 | 出现 NaN / Inf、overflow 连续发生、loss 不可恢复、rank divergence、checkpoint resume 后立刻发散 | 停止继续污染训练轨迹，冻结现场，进入根因定位流程 |

### 2.6 异常识别：不要只看“变大”，要看“变大的结构”
训练中很多指标都会波动。真正有诊断价值的不是单点数值，而是异常的结构。

#### 2.6.1 时间结构
| 时间现象 | 可能含义 |
| --- | --- |
| 从训练一开始就异常 | 初始化、mask、loss、precision、代码实现问题 |
| warmup 结束后异常 | LR 峰值过高、warmup 不足、optimizer state 尚未稳定 |
| resume 后立刻异常 | LR scheduler、optimizer state、loss scaler、ZeRO/FSDP shard 状态错误 |
| 某个数据阶段开始异常 | 数据分布突变、sequence length curriculum、source mix 改变 |
| 长时间缓慢恶化 | residual scale drift、weight norm drift、optimizer update 偏大 |
| 单 step 突发但可恢复 | 难样本、异常 batch、短期 precision overflow |
| 单 step 后不可恢复 | update 污染参数，可能需要回滚 checkpoint |


#### 2.6.2 空间结构
| 空间现象 | 可能含义 |
| --- | --- |
| 所有层一起异常 | LR、loss scale、optimizer、global batch、precision 问题 |
| 从某一层开始异常 | 该层 attention / MLP / norm / residual branch 问题 |
| 只有 attention head 异常 | QK score、mask、RoPE、head-specific parameter 问题 |
| 只有 MLP 异常 | activation function、gate/up/down projection、init 问题 |
| 只有 embedding / lm_head 异常 | tokenizer、vocab、tied weight、param group LR 问题 |
| 只有某 rank 异常 | distributed、data shard、overflow sync、FSDP/ZeRO 问题 |
| 只有某 expert 异常 | MoE routing、capacity、expert load、expert optimizer state 问题 |


#### 2.6.3 分布结构
| 分布现象 | 可能含义 |
| --- | --- |
| mean 正常但 p99 / max 极端 | 少数 outlier token、样本、head、expert 或参数行异常 |
| p50、p95、p99 整体抬升 | 全局尺度漂移，例如 LR、residual scaling、precision |
| histogram 被硬截断 | clip / clamp 正在改变真实分布 |
| heavy tail 逐步加重 | activation outlier 或 gradient outlier 正在积累 |
| top-k 样本贡献绝大部分 loss | 数据质量、label、sample weight、特殊 token 问题 |

### 2.7 根因与症状的区分标准
稳定性排查中最重要的判断是：某个异常是根因，还是下游症状。错误地把症状当根因，会导致长期依赖 clip / clamp；错误地把根因当普通波动，则会污染训练轨迹。

| 判断维度 | 更像根因 | 更像症状 |
| --- | --- | --- |
| 时间顺序 | 在 loss spike、NaN、grad explosion 前最早出现 | 在 loss spike 后才出现 |
| 空间定位 | 集中在特定 batch、layer、head、rank、expert 或模块 | 全模型同时异常，缺少局部来源 |
| 可复现性 | 固定 checkpoint + 固定 batch replay 稳定复现 | 随机出现，难以复现 |
| 机制解释 | 能解释从异常对象到训练发散的路径 | 只能说明“数值变大” |
| 干预效果 | 修复后下游异常消失，且不需要额外 hard clip | 压制后异常转移到其他对象 |
| 副作用 | 修复后训练质量不下降或恢复正常 | 压制后 loss 稳定但学习效率或 valid loss 变差 |


可以把证据强度分成三级：

| 证据等级 | 含义 | 示例 |
| --- | --- | --- |
| E0：相关性证据 | 某指标与 loss spike 同时出现 | grad norm spike 与 loss spike 同步 |
| E1：复现证据 | 固定 checkpoint 和 batch 可以复现 | replay 某个 packed batch 必炸 |
| E2：因果证据 | 修复该机制后异常消失 | 修复 packed boundary mask 后 replay 不再炸 |


只有达到 E2，才应将其认定为“根因已修复”。E0 和 E1 阶段可以临时使用保护机制，但不能把保护机制当成根因治理。

### 2.8 大模型稳定性训练的闭环系统
稳定性训练应被实现为一个闭环控制系统，而不是事后救火流程。

| 闭环阶段 | 输入 | 操作 | 输出 | 判断标准 |
| --- | --- | --- | --- | --- |
| 1. 建立基线 | 初始训练窗口、reference run、recipe 预期 | 记录 loss、activation、grad、update、precision、rank、batch 指标 | 正常范围和告警阈值 | baseline 覆盖 global / layer / rank / batch 维度 |
| 2. 持续监控 | 每 step 或每 N step 的统计 | 记录 mean、p95、p99、max、histogram、non-finite ratio | 时序指标和分布快照 | 能定位到 layer、head、expert、rank、batch |
| 3. 异常识别 | 当前指标与 baseline | 检测 NaN / Inf、spike、drift、outlier、rank divergence | 黄色 / 橙色 / 红色告警 | 告警能区分单点波动和持续异常 |
| 4. 异常归因 | checkpoint、batch、hook、rank log、optimizer state | 做 replay、ablation、precision switch、single-rank 对比 | 根因候选 | 至少定位到数据、架构、优化器、精度、分布式或代码类别 |
| 5. 根因修复 | 根因候选和最小复现实验 | 修 data、mask、LR、optimizer、architecture、precision、distributed state | 修复版本 | replay 不再异常，且不依赖新增 hard clip |
| 6. 保护干预 | 根因未知或短期难修 | 选择 norm / clip / clamp / rescale | 临时稳定训练 | 触发率低、范围小、可回滚、可审计 |
| 7. 干预验证 | 修复前后指标、质量指标、clip rate | 对比 loss、valid loss、update、entropy、downstream eval | 是否继续保留干预 | 保护机制没有长期重写训练动力学 |


闭环系统的关键要求是：**所有保护干预都必须可观测。**  
例如启用 gradient clipping 后，不能只记录 clipped 后的 gradient norm，还必须记录：

+ clipping 前的 global grad norm；
+ clipping 后的 global grad norm；
+ clip coefficient；
+ clip rate；
+ per-layer clip rate；
+ clip 前后 gradient direction cosine；
+ clip 触发时对应的 batch id、layer、rank、loss、update-to-weight ratio。

否则，clip 会把真正的异常分布隐藏起来。

### 2.9 稳定性问题的优先排查顺序
当训练出现异常时，建议按以下顺序排查。这个顺序的原则是：**先排除会使训练目标错误的语义问题，再排除会使实际 update 错误的优化问题，最后再考虑数值约束。**

#### Step 1：确认训练语义是否正确
优先检查：

| 对象 | 检查项 |
| --- | --- |
| label | 是否 shift 正确；padding 是否 ignore；特殊 token 是否参与 loss |
| loss | reduction 是否正确；per-token / per-sequence loss 是否符合预期 |
| tokenizer / vocab | vocab size、special token id、embedding resize 是否一致 |
| sequence packing | packed boundary 是否阻止跨样本 attention |
| attention mask | causal mask、padding mask、sliding window mask 是否正确 |
| position id | 长序列、packing、RoPE scaling 下是否正确 |


判断标准：  
如果 tiny batch 不能 overfit，或者手算一个 toy batch 的 loss 与实现不一致，应优先视为语义或代码问题，而不是稳定性超参问题。

#### Step 2：确认实际 update 是否正确
重点看：

[  
\frac{|\Delta W_l|}{|W_l|}  
]

而不是只看：

[  
|g_l|  
]

检查项包括：

| 对象 | 检查项 |
| --- | --- |
| LR | global LR、param-group LR、resume 后 scheduler step |
| warmup | warmup 是否足够；warmup 结束是否对应 spike |
| Adam state | (m)、(v)、step、bias correction、epsilon |
| weight decay | 是否错误作用于 norm、bias、embedding 等参数 |
| accumulation | loss scaling 和 gradient accumulation 是否等价于预期 batch |
| global batch size | tokens/update 是否与 LR scaling 匹配 |


判断标准：  
如果 gradient norm 没有明显异常，但 update norm 或 update-to-weight ratio 异常，根因更可能在 optimizer state、LR、epsilon、weight decay 或 resume 状态，而不是 forward activation。

#### Step 3：确认 forward signal 是否失控
检查：

| 对象 | 检查项 |
| --- | --- |
| residual stream | RMS 是否随 depth 或 step 持续上升 |
| attention scores | QK score p99 / max 是否异常 |
| attention entropy | 是否 collapse 到接近 one-hot，或过度均匀 |
| MLP activation | gate / up / down projection 是否出现 outlier |
| logits | max、RMS、entropy、top token 是否异常 |
| norm 层 | LayerNorm / RMSNorm input RMS 和 gamma 是否异常 |


判断标准：  
如果某层 residual RMS 开始阶跃式升高，应定位该层 attention output 和 MLP output，而不是直接对全模型 activation clamp。

#### Step 4：确认 precision 和 kernel 是否可靠
检查：

| 对象 | 检查项 |
| --- | --- |
| dtype | fp16 / bf16 / fp32 autocast 是否符合预期 |
| loss scaling | loss scale 是否频繁下降；overflow 是否连续发生 |
| softmax / CE | 是否存在 exp overflow、log underflow、除零、sqrt 负数 |
| fused kernel | FlashAttention、fused optimizer、fused CE 是否与 reference 实现一致 |
| non-finite | 第一个 NaN / Inf 出现在 forward 还是 backward |


判断标准：  
如果同一 checkpoint + batch 在 fp32 下稳定，在 fp16 下不稳定，应优先处理 precision、loss scaling 或 kernel，而不是把问题归因于数据。

#### Step 5：确认 distributed state 是否一致
检查：

| 对象 | 检查项 |
| --- | --- |
| rank-wise loss | 是否某个 rank loss 异常 |
| rank-wise grad | all-reduce 前后 grad norm 是否一致 |
| checkpoint | FSDP / ZeRO shard state 是否完整 |
| optimizer shard | optimizer state 是否按参数正确映射 |
| batch shape | 各 rank sequence length、tokens、mask 是否一致 |
| overflow sync | loss scaler overflow 是否在 rank 间同步处理 |


判断标准：  
如果单卡或少卡稳定，多卡不稳定，优先怀疑 distributed consistency，而不是模型本身不稳定。

### 2.10 找不到根因时如何选择 norm / clip / clamp / rescale
找不到根因时可以使用保护机制，但选择原则必须明确。保护机制的目标不是“把所有异常值压平”，而是**在最小改变训练动力学的前提下，阻止异常扩散并保留定位信息**。

#### 2.10.1 选择原则
| 原则 | 说明 | 例子 |
| --- | --- | --- |
| 靠近异常源头 | 在最接近异常产生位置的对象上施加约束 | attention score 异常优先查 Q/K norm 或 score softcap，而不是直接 clip 全模型梯度 |
| 优先平滑约束 | 优先使用 soft-capping、normalization、rescale，少用 hard clamp | logit soft cap 通常比 hard logit clipping 更可控 |
| 优先相对尺度 | 优先约束 update-to-weight ratio、grad-to-weight ratio 等相对量 | 某小范数层 update 过大时，global grad clip 未必足够 |
| 最小作用域 | 能 layer-wise 不 global，能 head-wise 不 layer-wise，能 expert-wise 不全模型 | 只有某 expert 异常时，不应无差别压制所有 expert |
| 可观测触发率 | 每次触发都记录原因、位置和幅度 | 记录 clip coefficient、pre/post norm、batch id |
| 可回滚 | 保护机制不应永久污染实验结论 | 保留启用保护前 checkpoint 和 replay batch |
| 不替代根因定位 | 高频触发时必须继续定位 | clip rate 长期升高时不能视为训练已稳定 |


#### 2.10.2 按异常对象选择保护机制
| 异常对象 | 优先保护机制 | 不建议优先使用 | 原因 |
| --- | --- | --- | --- |
| global gradient norm | global grad norm clipping | per-value clipping | global norm clipping 对方向破坏较小 |
| 某层 update-to-weight ratio | layer-wise update cap、降低该 param group LR | 盲目降低全局 LR | 全局降 LR 可能牺牲其他正常层学习 |
| attention score | QK Norm、temperature、score softcap | hard attention score clamp | hard clamp 容易破坏 attention 排序结构 |
| logits | logit soft-capping、lm_head scale 检查 | 直接 hard clipping 所有 logits | hard clipping 会影响概率校准和 CE 梯度 |
| activation outlier | residual / MLP scaling、局部 activation clamp | 全模型 hidden clamp | 全局 clamp 容易掩盖上游尺度漂移 |
| fp16 overflow | dynamic loss scaling、bf16、FP32 softmax | 忽略 overflow 或连续 skip step | overflow 是数值范围问题，需要显式处理 |
| MoE expert 梯度异常 | expert-wise grad clip、router z-loss、capacity 调整 | 全模型 grad clip 作为唯一手段 | expert 异常通常与 routing/load 有关 |
| reward outlier | reward normalization / clipping | 不查 reward model 和 prompt 分布 | reward clipping 会改变 RL 目标 |

### 2.11 如何判断 norm / clip 是否掩盖了真正问题
norm / clip / clamp / rescale 可以作为稳定性机制，但必须持续判断它是否正在掩盖根因。以下信号出现时，应将保护机制从“保险丝”重新判定为“可能掩盖问题的主训练机制”。

| 信号 | 具体判断标准 | 可能含义 | 应采取的动作 |
| --- | --- | --- | --- |
| clip rate 长期较高 | 连续多个窗口内持续触发，而不是偶发触发 | 梯度或 update 的自然尺度已经超过阈值 | 降 LR、查 optimizer state、查数据 batch、查 layer-wise 来源 |
| pre-clip norm 持续上升 | clip 后稳定，但 clip 前越来越大 | 上游异常仍在恶化 | 记录 pre-clip 分布，定位 first exploding layer |
| clipping coefficient 长期很小 | 大量 step 被强缩放 | clip 正在重写训练步长 | 调整 LR / batch / optimizer，而不是继续依赖 clip |
| clip 前后方向 cosine 低 | clipping 明显改变 gradient direction | 训练方向被保护机制扭曲 | 分层定位，避免全局粗暴 clip |
| 某层 / 某 expert 高频触发 | 触发集中在局部模块 | 局部模块或 routing 存在根因 | 查该层输入输出、param group、expert load |
| loss 稳定但 valid loss 变差 | 数值稳定但泛化或学习质量下降 | 保护机制改变了有效目标或优化路径 | 做 ablation，比较关闭保护后的短程 replay |
| upstream 指标继续恶化 | residual RMS、logits max、attention score 仍增长 | clip 只压住下游 gradient | 回到 forward signal 排查 |
| 换 fp32 / 降 LR 后 clip rate 消失 | 保护机制触发与精度或 LR 强相关 | 原因可能不是数据本身 | 修 precision 或 LR，而非永久 clip |
| histogram 出现硬边界 | 分布被截断在阈值处 | clamp / clipping 过强 | 使用 soft constraint 或修根因 |
| bad batch quarantine 比例上升 | 越来越多 batch 被跳过或降权 | 数据或 sampler 存在系统性问题 | 对数据源、长度、template、权重做审计 |


判断是否“掩盖问题”的核心标准是：  
**关闭或放松保护机制后，异常是否快速复现；修复候选根因后，保护机制触发率是否显著下降。**

如果保护机制触发率下降，但 loss 和 validation quality 不下降，说明根因修复有效；如果保护机制触发率下降但质量变差，需要判断原保护机制是否已经成为训练 recipe 的一部分，而不是单纯保险丝。

### 2.12 稳定性干预的 A / B / C 接口
本章中的定义需要与全文的 A / B / C 分类一致。

| 类型 | 在本章闭环中的位置 | 判断标准 | 例子 |
| --- | --- | --- | --- |
| A 类：根因修复 | 异常归因后的首选处理 | 找到具体机制并修复；修复后不再依赖额外 hard clip / clamp | 修 mask、修 label shift、恢复 optimizer state、修 LR scheduler、修 FSDP shard load |
| B 类：数值约束 | 根因未知、难修或需要保险时使用 | 主要通过限制幅度、归一化、裁剪或重缩放防止异常扩散 | global grad clipping、activation clamp、logit clipping、update norm cap |
| C 类：混合机制 | 可能是架构 recipe，也可能是止血手段 | 取决于使用动机、触发方式、是否长期作为模型设计、是否有机制解释 | RMSNorm、QK Norm、residual scaling、loss scaling、logit soft-capping、router z-loss |


需要特别强调：  
**C 类方法不能因为含有 norm、scaling 或 constraint 就被简单归为“治标不治本”。**  
例如 RMSNorm、Pre-LN、QK Norm、residual scaling 如果从模型设计开始就作为稳定训练 recipe 的一部分，并且配套初始化、LR、optimizer 和 depth scaling，那么它们更接近架构稳定性设计；如果是在训练发散后临时加入，用来压制某个未解释的异常，则更接近 B 类数值止血。

### 2.13 典型工程案例
#### 案例 1：resume 后 loss spike
现象：

+ resume 前 loss 平稳；
+ resume 后 1–5 step 内 loss spike；
+ global grad norm 升高；
+ clip 后训练暂时不 NaN。

优先判断：

| 检查对象 | 判断 |
| --- | --- |
| LR scheduler step | 是否从错误 step 重启 |
| Adam (m, v) | 是否丢失或 shape 映射错误 |
| loss scaler | 是否恢复到错误 scale |
| ZeRO / FSDP shard | optimizer shard 是否完整 |
| global step | logging step 与 optimizer step 是否一致 |


结论：  
这种情况下，gradient clipping 只是防止坏 update 扩散。真正的稳定性修复是恢复正确的 scheduler、optimizer state、loss scaler 和 shard mapping。

---

#### 案例 2：长序列阶段 attention entropy collapse
现象：

+ 进入 long-context 数据阶段后 loss spike 增多；
+ attention score p99 上升；
+ 部分 head 的 attention entropy 接近 0；
+ logits max 同步变大。

优先判断：

| 检查对象 | 判断 |
| --- | --- |
| position id | packing 后是否连续且符合 RoPE 预期 |
| RoPE scaling | 长度外推配置是否正确 |
| attention mask | 是否存在跨样本 attention 或非法 future attention |
| Q/K norm | 长序列下是否持续增长 |
| score distribution | 是所有 head 异常还是少数 head 异常 |


结论：  
如果根因是 position id 或 mask 错误，必须按 A 类修复。  
如果机制上是 QK score 在长序列下系统性过大，则 QK Norm、temperature 或 attention score softcap 可能属于 C 类稳定设计，而不是简单“退烧药”。

---

#### 案例 3：grad norm 正常但 update-to-weight ratio 异常
现象：

+ global grad norm 没有明显 spike；
+ 某些层的 update-to-weight ratio 明显偏大；
+ loss 在数百 step 内缓慢恶化；
+ global grad clipping 基本不触发，但训练仍发散。

优先判断：

| 检查对象 | 判断 |
| --- | --- |
| Adam (v) | 是否异常偏小 |
| epsilon | 是否导致 effective update 放大 |
| param-group LR | 某些层 LR 是否配置错误 |
| weight decay | 是否错误作用于 norm / bias / embedding |
| optimizer state | resume 后 state 是否错配 |


结论：  
这类问题说明只看 gradient norm 不够。稳定性闭环必须监控 optimizer update 和 update-to-weight ratio。修复方向通常是 optimizer state、LR group、epsilon、weight decay 或 schedule，而不是优先增加 gradient clipping。

---

#### 案例 4：MoE 某些 expert 梯度异常
现象：

+ 全局 loss 偶发 spike；
+ 某几个 expert 的 token load 和 grad norm 明显高于其他 expert；
+ router entropy 降低；
+ dropped token ratio 升高。

优先判断：

| 检查对象 | 判断 |
| --- | --- |
| router logits | 是否过大或过尖 |
| expert load | 是否长期不均衡 |
| capacity factor | 是否导致大量 token drop |
| aux loss / z-loss | 权重是否过低或过高 |
| expert optimizer state | 是否某些 expert state 错配 |
| data source | 是否某类样本集中路由到少数 expert |


结论：  
如果只是对全模型做 global grad clipping，可能掩盖 expert-level routing 问题。更合理的闭环是先定位 router / capacity / expert load，再决定是否使用 expert-wise clipping 或 router z-loss 作为保护机制。

### 2.14 本章的最终定义
综合以上内容，可以将“大模型稳定性训练”定义为：

**大模型稳定性训练，是对 LLM 训练过程中训练语义、信号传播、优化动力学、数值精度和分布式系统状态进行闭环控制，使 loss、logits、attention scores、hidden states、activation、gradient、parameter、optimizer state、optimizer update、learning rate、batch distribution、sequence / mask、precision、rank consistency 和 MoE routing 始终处于可学习、可解释、可复现、可恢复的动态范围内。其工程原则是：优先通过监控和最小复现实验定位导致异常的具体机制并修复根因；当根因未知、短期难修或需要冗余保护时，使用最小扰动的 norm / clip / clamp / rescale 作为保险丝；同时持续监控这些保险丝的触发率、作用范围、方向扭曲和质量影响，防止它们掩盖数据、代码、优化器、架构、精度或分布式系统中的真实问题。**

一句话概括：

**稳定性训练的目标不是压小所有数值，而是保证训练轨迹中的语义正确、信号可传、梯度可用、更新可控、数值有限、系统一致；根因修复是治理，norm / clip 是保险，架构归一化是设计，优化器约束是训练动力学的一部分。**

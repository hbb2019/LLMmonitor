本章给出一个面向 LLM 预训练、SFT、RLHF、MoE 和分布式混合精度训练的工程化诊断流程。该流程延续全文的核心闭环：**监控 → 异常检测 → 时序/空间/因果归因 → 最小验证实验 → 根因修复 → 必要时使用 norm / clip / clamp / rescale 止血 → 验证止血手段没有掩盖真实问题**。

诊断的目标不是“把所有异常指标压小”，而是回答四个问题：

1. **第一个异常对象是什么？**  
是 loss、logits、attention score、activation、gradient、optimizer update、optimizer state、rank state，还是 batch 数据本身？
2. **异常发生在哪个阶段？**  
是 dataloader / forward / backward / optimizer step / checkpoint resume / distributed sync / MoE routing 中的哪一个阶段？
3. **异常是局部还是全局？**  
是单个 batch、单个 rank、单层、单个 head、单个 expert，还是全模型同步恶化？
4. **干预是否具有因果性？**  
replay、降低 LR、切换精度、禁用 fused kernel、去掉异常样本、重置 optimizer state、单卡复现等最小实验，是否能稳定改变异常结果？

---

### 7.1 总体排查顺序
稳定性诊断应遵循“先冻结现场，再全局分流，再局部定位，最后做最小因果验证”的顺序。

```plain
异常触发
  ↓
冻结现场：checkpoint / batch / RNG / optimizer / scaler / scheduler / rank state
  ↓
全局分流：loss / grad / update / logits / overflow / rank divergence
  ↓
阶段定位：data → forward → backward → optimizer step → distributed sync → checkpoint
  ↓
空间定位：batch / token / layer / head / expert / rank / param group
  ↓
最小验证实验：replay / ablation / precision switch / LR change / single-rank / state reset
  ↓
根因修复：数据、mask、LR、optimizer、架构、精度、分布式、MoE、代码
  ↓
必要止血：clip / norm / clamp / rescale
  ↓
回归验证：坏 batch 稳定、正常 batch 不退化、clip rate 不高频、valid loss 不恶化
```

这一流程有三个关键原则。

**第一，优先定位“第一个异常状态转移”。**  
不要只看最终的 NaN / Inf。NaN 往往是末端症状，真正的源头可能是更早的 attention score 过大、residual RMS 抬升、Adam update 异常、某个 rank 的 batch mask 错误，或者 checkpoint resume 后 scheduler step 错误。

**第二，把 forward、backward 和 optimizer update 分开看。**  
很多问题在 loss 和 gradient 层面看起来相似，但根因不同。例如：

| 现象 | 更可能的问题 |
| --- | --- |
| forward 里 logits / attention score 已异常 | 数据、mask、position、activation、架构、精度 |
| backward 后 grad norm 异常，但 activation 正常 | loss 定义、label、sample weight、梯度传播、局部模块 |
| grad norm 正常但 update norm 异常 | Adam state、epsilon、beta、LR、weight decay、resume state |
| optimizer step 后下一步开始异常 | update 过大、参数组配置、optimizer state、checkpoint 恢复 |
| 单 rank 异常但其他 rank 正常 | 分布式数据、rank-local overflow、all-reduce、FSDP/ZeRO shard |
| 只有某些 expert 异常 | MoE routing、capacity、aux loss、expert state |


**第三，所有修复必须经过 replay 或对照实验。**  
只要异常可复现，就应该尽量构造最小验证实验。不能只根据一次 loss 下降判断修复成功。修复后至少要验证：

+ 同 checkpoint + 同 batch replay 是否稳定；
+ 同一时间窗口内正常 batch 是否不退化；
+ grad / update / activation / logits 的分布是否回到稳定窗口；
+ clip、overflow、skip step、bad batch quarantine 的触发率是否下降；
+ valid loss 或 held-out loss 是否没有异常恶化。

---

### 7.2 异常触发等级与立即动作
训练稳定性告警可以按严重程度分级处理。分级的目的不是机械停训，而是决定是否允许继续 optimizer step，以及需要保存哪些现场信息。

| 等级 | 触发条件 | 典型信号 | 立即动作 | 是否允许继续训练 |
| --- | --- | --- | --- | --- |
| P0：非有限值 | 出现 NaN / Inf / overflow 后无法恢复 | loss NaN、grad NaN、param NaN、optimizer state NaN | 立即停止当前 step；禁止写入污染 checkpoint；保存异常前 checkpoint、batch、rank 日志；定位 first non-finite tensor | 不允许直接继续 |
| P1：强异常 | loss spike、grad norm、update norm、logits max 相比稳定窗口 p95 放大 5–10 倍以上，或连续多 step 恶化 | loss 连续 spike、clip rate 激增、update-to-weight ratio 异常 | 暂停训练；从最后健康 checkpoint replay；做全局分流 | 通常不建议继续 |
| P2：趋势性漂移 | residual RMS、attention entropy、router entropy、expert load、loss scale 等持续偏离 | 指标未 NaN，但趋势单调恶化 | 继续短窗口监控；提高日志粒度；准备局部 hook；必要时降低 LR 做对照 | 可短时间继续，但需验证 |
| P3：弱告警 | 单 batch 高 loss、少量 overflow、低频 clip | 难样本、长样本、偶发 outlier | 记录样本和指标；不立即处理；进入统计观察 | 可继续 |


阈值不应使用全局固定常数，而应基于**稳定参考窗口**建立。参考窗口一般取最近一段健康训练区间，例如 warmup 后 loss 平稳的 100–1000 个 step。推荐使用：

+ median / p95 / p99；
+ moving average 与 moving std；
+ 按层、按 head、按 expert、按 rank 的相对比值；
+ 与相邻层、相邻 step、相同数据源 batch 的对比。

工程上可以使用如下经验规则：

| 指标变化 | 判断 |
| --- | --- |
| 单个指标相对稳定窗口 median 放大 2–3 倍 | 弱告警，继续观察或增加日志 |
| 相对 p95 放大 3–5 倍 | 进入诊断，保存 batch 和局部指标 |
| 相对 p95 放大 5–10 倍以上 | 强异常，建议 replay |
| 出现 NaN / Inf | P0，必须定位 first non-finite |
| clip rate 长期超过低频保险丝水平 | 说明 clip 可能正在参与定义训练动力学 |
| 某层指标长期显著高于相邻层 5–10 倍 | 局部模块、初始化、mask、LR group 或 update 问题优先排查 |
| rank 间 loss / grad / overflow 差异长期异常 | 分布式或 rank-local 数据问题优先排查 |

### 7.3 第一步：冻结现场，而不是立刻调 clip
出现异常后，最容易犯的错误是直接降低 LR、加大 clip 或跳过 batch。这样可能让训练暂时恢复，但会丢失根因证据。正确做法是先保存足够的诊断上下文。

| 需要保存的对象 | 具体内容 | 诊断目的 |
| --- | --- | --- |
| 模型 checkpoint | 异常前最近健康 checkpoint、异常 step 前后参数 | 判断异常是否由某个 update 引入 |
| optimizer state | Adam m/v、step count、param group、weight decay、epsilon、beta | 判断 update 是否异常、resume 是否错 |
| scheduler state | global step、warmup step、当前 LR、各参数组 LR | 判断 LR / warmup / resume 是否错 |
| loss scaler state | loss scale、overflow count、skipped step | 判断混合精度问题 |
| RNG state | Python / NumPy / PyTorch / CUDA RNG、dropout seed | 保证 replay 可复现 |
| dataloader state | sample id、source、length、token ids、labels、sample weight | 定位异常 batch 或样本 |
| mask / position | attention mask、padding mask、packed boundary、position id、RoPE index | 定位 sequence packing 和 mask 错误 |
| distributed state | rank id、rank-local batch、rank loss、rank grad norm、param checksum | 定位 rank divergence |
| MoE state | router logits、expert load、dropped tokens、aux loss、z-loss、expert grad | 定位 expert collapse 或 routing 问题 |
| kernel / precision 配置 | autocast、bf16/fp16、FlashAttention、fused optimizer、通信 dtype | 定位数值和 kernel 问题 |


冻结现场时有三条硬规则：

1. **如果参数或 optimizer state 已经出现 NaN，不要继续写入主 checkpoint。**  
应保存为隔离的 debug checkpoint，避免污染后续恢复点。
2. **不要只保存均值。**  
大模型训练中很多异常是 p99、max、outlier ratio、某个 rank 或某个 expert 的问题，均值会掩盖它。
3. **不要只保存异常 step。**  
至少保存异常前若干 step 的指标轨迹。稳定性诊断需要看异常的先后顺序，而不是只看终点。

---

### 7.4 第二步：全局指标分流
全局分流的目的，是快速判断异常优先属于哪条诊断分支：数据、优化、架构、精度、分布式、MoE 或代码实现。

首先检查以下全局指标：

| 指标 | 重点看什么 | 典型解释 |
| --- | --- | --- |
| train loss | 当前值、moving average、spike ratio、与上一 step 差值 | 判断是否出现训练目标层面的异常 |
| valid / held-out loss | 是否同步恶化 | 判断是训练 batch 特异问题还是模型整体退化 |
| logits max / RMS / entropy | logits 是否饱和，top-1 prob 是否过高 | 判断输出层或 hidden state 是否失控 |
| global grad norm | norm、p95、max、non-finite | 判断 backward 是否异常 |
| global update norm | Adam / optimizer 实际更新量 | 判断 optimizer step 是否异常 |
| update-to-weight ratio | ||ΔW|| / ||W||，按参数组统计 | 判断相对参数尺度是否过大 |
| LR / warmup / schedule | 当前 LR、global step、param group LR | 判断 scheduler 或 resume 错误 |
| optimizer state | Adam m/v finite ratio、step count、v min/max | 判断 state 是否损坏或尺度异常 |
| loss scale / overflow | loss scale、overflow count、skip step | 判断 FP16 / AMP 问题 |
| rank-wise loss / grad | rank 间差异 | 判断分布式或 rank-local 数据问题 |
| clip rate | global/layer/update clip 触发率 | 判断是否依赖止血机制 |
| tokens per update | 实际 token batch、accumulation、global batch | 判断 effective batch 和 LR scale 是否一致 |


全局指标的分流规则如下。

| 全局现象 | 优先怀疑 | 下一步操作 |
| --- | --- | --- |
| resume 后第一个或前几个 step 立即 spike | scheduler step、LR、optimizer state、loss scaler、ZeRO/FSDP shard | 对比 resume 前后 LR、Adam step、m/v shape、param checksum；从上一个 checkpoint replay |
| loss spike，但 grad/update 之前已有 logits 极端 | 数据、mask、position、hidden / lm_head 尺度 | 查 per-sample loss、logits token outlier、mask、position id |
| grad norm 爆，但 loss/logits 没明显先兆 | loss reduction、label、sample weight、backward kernel、局部模块 | 分层 grad hook，检查 loss 标量缩放和 label mask |
| grad norm 正常但 update norm 或 U/W 异常 | optimizer state、epsilon、beta、LR group、weight decay | 记录 m/v 和 effective update；做 reset optimizer state 对照 |
| update 后下一步开始 activation/logits 异常 | 上一步 update 过大或参数组配置错 | 比较 step 前后 param delta；定位异常 param group |
| fp16 overflow 频繁，bf16/fp32 对照稳定 | 精度范围、loss scaling、softmax / exp / fused kernel | 切换 bf16/fp32 单步 replay；关闭 fused kernel |
| 单卡稳定，多卡不稳定 | DDP/FSDP/ZeRO、accumulation、rank-local batch、all-reduce | 单卡/少卡复现；记录 all-reduce 前后 grad |
| 某 rank 的 loss/grad/overflow 异常 | rank-local 数据、sampler、通信、shard state | 保存该 rank batch；检查 param checksum |
| MoE loss spike 伴随 expert load collapse | router logits、capacity factor、aux loss、z-loss、expert state | 查 router entropy、expert load、dropped tokens |
| clip rate 长期高但 loss 未 NaN | clip 可能正在掩盖 LR、数据或架构问题 | 查看 clip 前分布、clip 后方向 cosine、上游指标是否仍恶化 |


---

### 7.5 第三步：定位异常发生阶段
全局分流后，应把一次训练 step 拆成多个阶段分别检查。

```plain
batch load
  → tokenization / packing / mask / position
  → forward
  → loss computation
  → backward
  → gradient synchronization
  → gradient clipping
  → optimizer update
  → parameter synchronization / sharding
  → scheduler / scaler update
  → next step
```

不同阶段对应的诊断重点不同。

| 阶段 | 关键检查 | 异常含义 |
| --- | --- | --- |
| batch load | sample id、length、source、label、weight、special token | 数据、采样、权重、template 问题 |
| packing / mask | causal mask、padding mask、packed boundary、position id | sequence 语义错误 |
| forward | residual RMS、attention score、logits、activation finite | 架构、mask、数据、精度问题 |
| loss computation | label shift、ignore_index、loss reduction、per-token loss | 训练目标或 loss 实现问题 |
| backward | grad norm、layer grad、non-finite grad | 梯度传播或 loss scale 问题 |
| gradient sync | all-reduce 前后 grad、rank-wise norm | 分布式通信或 rank 数据问题 |
| clipping | clip 前后 norm、clip rate、direction cosine | 止血机制是否过强 |
| optimizer update | m/v、update norm、U/W、param delta | optimizer state、LR、weight decay 问题 |
| scheduler / scaler | LR step、loss scale、overflow skip | resume、AMP、warmup 问题 |
| next step | 参数更新后 activation 是否恶化 | 上一步 update 是否引入不稳定 |


关键判据是：**异常第一次出现在哪个阶段，就优先从该阶段向上游找根因。**

例如：

+ forward 中 attention score 已经极端，优先查 Q/K norm、mask、RoPE、precision；
+ backward 后才出现 NaN，优先查 loss scale、gradient kernel、loss reduction；
+ optimizer step 后参数变 NaN，优先查 optimizer state、LR、update clipping、weight decay；
+ all-reduce 前各 rank 正常，all-reduce 后异常，优先查通信 dtype、bucket、FSDP/ZeRO 状态。

---

### 7.6 第四步：layer-wise 诊断
layer-wise 诊断的核心是找到“第一个坏层”或“第一个坏模块”。不要只看最后一层，也不要只看全局 norm。

建议在每个 Transformer block 记录：

| 模块 | 监控对象 |
| --- | --- |
| block input | residual RMS、p95/p99、max、finite ratio |
| attention Q/K/V | Q norm、K norm、V norm、head-wise RMS |
| attention score | score max、p99、entropy、max attention prob、illegal attention ratio |
| attention output | RMS、p99、grad norm |
| MLP gate/up/down | activation RMS、max、kurtosis、down_proj output RMS |
| norm 层 | LN/RMSNorm input RMS、output RMS、gamma norm |
| block output | residual RMS、block output/input ratio |
| 参数 | param norm、grad norm、update norm、U/W ratio |
| optimizer state | layer-wise Adam m/v、effective update |


layer-wise 判断规则如下。

| 现象 | 判断标准 | 优先怀疑 |
| --- | --- | --- |
| 从某一层开始 residual RMS 阶跃上升 | 该层输出正常输入的数倍，且后续层继承异常 | 该层 attention 或 MLP 输出异常 |
| residual RMS 随深度缓慢单调上升 | 多层累计，而非单点异常 | residual scaling、Post-LN 深层不稳、LR 过大 |
| 某层 attention score p99 极端 | Q/K norm 同步变大或 entropy collapse | QK scale、mask、RoPE、temperature |
| 某层 MLP down_proj 输出极端 | gate/up activation heavy tail | activation、init、MLP scale |
| layer grad norm 异常但 forward activation 正常 | backward 路径、loss、局部参数组、梯度同步 | loss reduction、grad scaling、DDP |
| layer update norm 异常但 grad norm 正常 | optimizer state 或 LR group | Adam v 太小、epsilon、param group 错 |
| 某层 U/W 长期高于其他层 | 相对更新过大 | 层级 LR、weight decay、初始化、optimizer state |
| LN/RMSNorm gamma 快速增大 | norm 层在补偿上游尺度问题 | residual/attention/MLP 尺度失衡 |


定位时应使用三条规则。

**第一，input-normal / output-abnormal 规则。**  
如果某层输入正常、输出异常，该层是重点嫌疑对象。  
如果某层输入已经异常、输出只是继承异常，应继续向上游找。

**第二，forward-first / backward-first 规则。**  
forward 先异常，优先查数据、mask、架构、precision。  
backward 才异常，优先查 loss、gradient scaling、optimizer、distributed sync。

**第三，local-vs-depth 规则。**  
孤立层异常更像局部 bug、参数组、mask、某个 head/expert 问题。  
随深度累积异常更像 residual scaling、Pre-LN/Post-LN、LR、初始化或 update 尺度问题。

---

### 7.7 第五步：batch / token / sequence 诊断
如果异常与特定 batch 高度相关，应从 batch 维度继续缩小范围。

建议记录以下对象：

| 对象 | 诊断指标 | 可能发现的问题 |
| --- | --- | --- |
| sample | sample id、source、domain、language、timestamp | 某数据源或某类型样本触发 spike |
| sequence | length、padding ratio、packed segment count | 极长样本、packing 异常 |
| token | special token ratio、UNK/OOV、重复 n-gram | tokenizer/template 异常 |
| label | ignore ratio、label shift、label range | padding 未 mask、错位 label |
| loss | per-token loss、per-sequence loss、top-k loss token | 找到异常 token 或样本 |
| weight | sample weight、loss weight | 样本权重错误 |
| mask | attention mask density、非法 attention count | causal/padding/packed boundary 错误 |
| position | position id range、RoPE index、reset position | position id 或 RoPE scaling 错误 |


推荐的 batch 诊断流程：

```plain
异常 batch replay
  ↓
检查是否稳定复现
  ↓
若复现：计算 per-sequence loss / per-token loss
  ↓
按样本二分：bad half / good half
  ↓
定位 top-loss 样本或异常 packed segment
  ↓
检查 length / source / special token / label / mask / position
  ↓
去掉异常样本或修正 mask 后 replay
  ↓
若异常消失，确认数据或 sequence 处理根因
```

几个高价值检查：

| 检查 | 具体做法 | 判断标准 |
| --- | --- | --- |
| bad batch replay | 固定 checkpoint、seed、batch，重复 forward/backward | 若稳定复现，batch 或该状态高度相关 |
| batch 二分 | 将 batch 分成两半递归 replay | 找到最小触发样本集合 |
| per-token loss top-k | 输出 loss 最高 token 及其上下文 | 若集中在 pad、special token、边界 token，查 label/mask |
| packed vs unpacked | 同样样本分别 packed/unpacked 训练 | packed 炸而 unpacked 不炸，优先查 boundary mask |
| short vs long | 按 length 分桶 replay | 长序列炸，查 position/RoPE/attention precision |
| source ablation | 去掉某数据源或降低权重 | 若异常消失，查该源数据质量或采样权重 |
| tiny batch overfit | 用极小干净 batch 训练到低 loss | 若不能 overfit，查 loss/label/model 实现 |
| hand-check CE | 对一个 toy batch 手算 cross entropy | 验证 label shift、ignore_index、loss reduction |


需要注意：单个 batch 高 loss 不必然是错误。判断是否需要处理，应看以下标准：

+ 是否可稳定复现 spike；
+ 是否引起 grad/update/activation 连续恶化；
+ 是否集中在明显错误 token、padding、packed boundary 或非法 label；
+ 是否来自异常 source 或异常 sample weight；
+ 去掉该 batch 后训练是否恢复，且正常 batch 不受影响。

### 7.8 第六步：分布式与 MoE 专项诊断
#### 7.8.1 分布式训练诊断
分布式问题经常表现为数值不稳定，但根因是状态不一致。需要区分 all-reduce 前、all-reduce 后、optimizer step 前、optimizer step 后的状态。

| 现象 | 重点检查 | 最小验证实验 | 可能修复 |
| --- | --- | --- | --- |
| 单卡稳定，多卡不稳定 | rank batch、loss reduction、gradient accumulation、DDP sync | 用相同 global batch 做单卡/多卡等价对比 | 修 accumulation scaling、no_sync、loss averaging |
| 某 rank loss 异常 | rank-local sample、mask、position、source | 保存该 rank batch 单独 replay | 修 sampler、数据切分、rank-local preprocessing |
| all-reduce 前正常，后异常 | 通信 dtype、bucket、overflow 同步 | 记录 all-reduce 前后 grad norm | 修通信 dtype、DDP bucket、finite check |
| resume 后某些 rank 异常 | FSDP/ZeRO shard、optimizer state、param checksum | full-state checkpoint 对比 sharded checkpoint | 修 checkpoint save/load、reshard 逻辑 |
| optimizer shard NaN | shard state finite ratio、Adam m/v shape | 禁用 ZeRO/FSDP 小规模 replay | 回滚 checkpoint，修 state dict |
| loss scale 不一致 | rank-wise overflow、scaler state | 强制同步 overflow 和 scaler | 修 AMP scaler 同步 |
| global batch 不符 | tokens/update、microbatch、accumulation step | 打印实际 tokens per update | 修 batch size 和 LR scaling |


推荐记录：

+ all-reduce 前后 grad norm；
+ 每个 rank 的 loss、overflow、clip rate；
+ 每个 rank 的 batch shape、seq length、sample source；
+ 参数 checksum 或关键层 norm；
+ optimizer shard finite ratio；
+ accumulation step 与实际 tokens/update。

如果 rank 间参数已经不一致，继续训练通常没有诊断价值。应回滚到最后一致 checkpoint，并检查 checkpoint round-trip。

#### 7.8.2 MoE 诊断
MoE 训练的稳定性问题通常来自 routing、expert load、capacity、router logits、aux loss 或 expert optimizer state。

| 现象 | 重点检查 | 最小验证实验 | 可能修复 |
| --- | --- | --- | --- |
| expert load collapse | tokens/expert、load CV、top expert share | 固定 batch 记录 router top-k 分布 | 调 aux load balancing loss、router temperature、router init |
| router logits 爆炸 | router logit max/RMS、router entropy、z-loss | 加/调 router z-loss 或降低 router LR 对照 | router z-loss、router precision、router LR |
| dropped tokens 高 | capacity usage、drop ratio、seq length | 调 capacity factor replay | 调 capacity factor、batching、routing policy |
| 某 expert grad 爆 | per-expert grad norm、expert load、expert data | 固定 routing 或禁用该 expert 对比 | expert LR、expert clip、router balance |
| 某 expert update 大 | per-expert U/W、optimizer state | 检查 expert Adam m/v | 修 expert optimizer state、LR group |
| aux loss 主导训练 | main loss vs aux loss 比例 | 降低 aux loss 权重对照 | 调 aux/z-loss 权重 |
| MoE 多卡不一致 | expert parallel dispatch、combine、capacity mask | 小规模 expert-parallel replay | 修 dispatch/combine、通信、padding |


MoE 中的止血手段，例如 expert grad clipping、router logit softcap、z-loss 增大，必须配合 expert load 和 task loss 监控。否则可能出现“数值稳定但路由退化”的情况：所有 token 被过度均匀分配，expert specialization 被破坏。

---

### 7.9 第七步：按症状进入专项排查路径
#### 7.9.1 loss spike
```plain
loss spike
  ↓
确认是否单 batch 触发
  ↓
查看 logits / activation / grad / update 谁先异常
  ↓
若 batch 相关：per-sequence / per-token / source / mask / position
  ↓
若全局相关：LR / optimizer / precision / distributed
  ↓
replay 验证
  ↓
根因修复或临时止血
```

| 排查项 | 判断标准 | 操作 |
| --- | --- | --- |
| 是否可 replay | 同 checkpoint + 同 batch 是否稳定 spike | 可复现则 batch/状态相关；不可复现则查非确定性、分布式、dropout、kernel |
| spike 是否集中在 top-loss 样本 | per-seq loss 是否少数样本极端 | 检查样本、label、source、sample weight |
| logits 是否先极端 | spike 前 logits max / entropy 是否异常 | 查 hidden RMS、lm_head、tokenizer、attention |
| grad 是否先爆 | loss 不极端但 grad norm 大 | 查 loss reduction、sample weight、backward |
| update 是否先异常 | grad 正常但下一步 loss spike | 查 Adam state、LR、U/W |
| valid loss 是否同步恶化 | 训练 batch spike 但 valid 正常 | 更像 batch 局部问题；valid 也恶化则模型状态已受污染 |


修复优先级：

1. 修数据、label、mask、sample weight；
2. 修 LR、scheduler、optimizer state；
3. 修 precision / kernel；
4. 修架构尺度；
5. 只有在根因短期不可定位时，使用 grad clipping、bad batch quarantine 或 loss clipping，并记录触发率。

#### 7.9.2 grad norm 快速变大
| 排查项 | 判断标准 | 操作 |
| --- | --- | --- |
| loss 是否先 spike | loss 先异常，grad 是下游症状 | 先查数据和 loss |
| activation 是否先异常 | forward hidden/attention/MLP 已变大 | 查架构、mask、precision |
| layer grad 是否局部异常 | 某层或某模块 grad 明显大于相邻层 | 做 layer hook，查该层输入/输出 |
| grad 是否 rank-local 异常 | 某 rank grad 大 | 查 rank batch 或 all-reduce |
| clip 前后方向是否变化大 | clipped grad 与原 grad cosine 低 | clip 过强，可能重写训练方向 |
| update 是否同步变大 | grad 大但 update 被 Adam 抑制，或 update 更大 | 查 optimizer state |


最小验证实验：

+ 降低 LR replay；
+ 用相同 batch 关闭 mixed precision replay；
+ 单卡 replay；
+ 去掉 top-loss 样本 replay；
+ 分层冻结或局部模块替换 replay；
+ 检查 loss reduction 是否按 token 而不是按 batch 错误缩放。

临时止血：

+ global grad norm clipping；
+ layer-wise grad clipping；
+ 对异常 expert 做 expert grad clipping；
+ 但如果 clip rate 持续较高，需要继续查 LR、数据、activation 或 optimizer 根因。

#### 7.9.3 activation norm 快速变大
| 排查项 | 判断标准 | 操作 |
| --- | --- | --- |
| 哪一层开始变大 | 找到 first bad layer | 检查该层 attention 和 MLP 分支 |
| attention 还是 MLP | attention output / MLP output 谁先异常 | 分别查 QK score 或 gate/up/down activation |
| residual 是否累积 | 所有深层逐步抬升 | 查 residual scaling、Pre-LN/Post-LN、LR |
| 是否 batch-specific | 只有长序列或某数据源触发 | 查 sequence length、mask、position |
| 是否 precision-specific | bf16/fp32 稳定，fp16 异常 | 查 softmax、matmul accumulation、loss scaling |


修复方式：

+ 修 attention mask、RoPE、position id；
+ 修初始化、residual branch scale、MLP activation；
+ 调整 LR / warmup；
+ 使用 QK Norm、residual scaling、Pre-LN 等结构性方法；
+ activation clamp 只作为短期保护，不能长期替代根因修复。

#### 7.9.4 logits 变得极端
| 排查项 | 判断标准 | 操作 |
| --- | --- | --- |
| final hidden RMS 是否大 | hidden RMS 大导致 logits 大 | 向上查 residual stream |
| lm_head norm 是否大 | hidden 正常但 logits 大 | 查 lm_head 初始化、LR group、tie embedding |
| 是否特定 token 极端 | 某些 token logit 长期异常 | 查 tokenizer、special token、数据模板 |
| attention entropy 是否 collapse | attention 过尖导致 hidden 异常 | 查 QK norm、attention score |
| label 是否错配 | loss 高且 logits 对错 token 过强 | 查 label shift、vocab 对齐 |


修复方式：

+ 修 tokenizer/vocab、special token、label shift；
+ 修 lm_head / embedding scale；
+ 修 hidden RMS 源头；
+ 使用 logit soft-capping 作为平滑保护；
+ 避免长期 hard clipping logits，因为它会改变校准和尾部概率结构。

#### 7.9.5 attention entropy 异常
attention entropy 需要区分两类异常。

| 类型 | 现象 | 优先怀疑 |
| --- | --- | --- |
| entropy collapse | attention 接近 one-hot，max attention prob 高 | Q/K norm 大、score scale 大、RoPE、temperature、mask |
| entropy 过高 | attention 接近均匀，head 不选择信息 | score scale 太小、QK 表达不足、temperature 过高、mask 错 |


排查流程：

1. 记录 head-wise entropy，而不是只看 layer mean；
2. 查看 attention score p99 / max；
3. 查看 Q norm、K norm；
4. 检查 mask 是否允许非法 attention；
5. 按 sequence length 分桶；
6. 比较短上下文和长上下文；
7. 如果是 MoE 或多模态模型，还要查看 modality-specific attention pattern。

修复方式：

+ 修 mask、position id、RoPE scaling；
+ 调整 attention temperature 或 score scaling；
+ 引入或调节 QK Norm；
+ 使用 attention score softcap；
+ hard clamp attention score 只能作为最后止血手段。

#### 7.9.6 optimizer update norm 或 update-to-weight ratio 异常
optimizer update 是很多稳定性问题的真正源头。只看 gradient norm 不够。

| 排查项 | 判断标准 | 操作 |
| --- | --- | --- |
| grad norm 是否正常 | grad 正常但 update 大 | 查 Adam v、epsilon、bias correction |
| U/W 是否局部异常 | 某层或参数组 U/W 高 | 查 param group LR、weight decay |
| resume 后是否异常 | step count、m/v、scheduler 不连续 | 修 checkpoint state |
| v 是否过小 | v min 或均值异常小 | 调 epsilon / beta2，检查 state |
| weight decay 贡献是否过大 | decay update 与 gradient update 比例异常 | 修 AdamW 配置和 param group |
| small-norm 参数是否高 U/W | norm/bias/embedding 层易异常 | 单独统计参数组 |


最小验证实验：

+ 从同 checkpoint 使用正确 optimizer state replay；
+ reset optimizer state replay；
+ 降低 LR replay；
+ 禁用 weight decay replay；
+ 调整 epsilon replay；
+ 对异常参数组单独降低 LR；
+ 比较 update 前后参数 norm 和 logits/activation。

修复方式：

+ 恢复正确 optimizer state；
+ 修 scheduler / global step；
+ 修 param group LR / weight decay；
+ 调 epsilon / beta；
+ 使用 update clipping 或 U/W cap 作为临时或算法级约束。

#### 7.9.7 出现 NaN / Inf
NaN / Inf 的定位目标是找到 first non-finite tensor。

```plain
NaN / Inf
  ↓
是否输入已有 NaN / Inf
  ↓
forward 中 first non-finite
  ↓
若 forward 正常，查 loss
  ↓
若 loss 正常，查 backward grad
  ↓
若 grad 正常，查 all-reduce
  ↓
若 all-reduce 正常，查 optimizer update
  ↓
若 update 正常，查 next step
```

| first non-finite 位置 | 优先怀疑 | 操作 |
| --- | --- | --- |
| input tensor | 数据预处理、token id、mask 值 | 检查 dataloader 和 preprocessing |
| attention score / softmax | QK score、mask、fp16 overflow | stable softmax、bf16/fp32、QK scale |
| MLP activation | activation outlier、init、precision | 查 gate/up/down，切精度 |
| loss | label、ignore_index、log/exp、loss reduction | toy batch 手算 |
| gradient | loss scale、backward kernel、grad overflow | AMP scaler、bf16/fp32、禁 fused kernel |
| all-reduce 后 | 通信、rank finite 状态 | all-reduce 前后对比 |
| optimizer state | Adam m/v、fused optimizer、epsilon | 禁 fused optimizer、检查 state |
| parameter after step | update 过大、LR、weight decay | 回滚 step，查 update norm |


处理原则：

+ 非有限值出现后不要继续写入正式 checkpoint；
+ 可以 skip 当前 step，但连续 skip 不是修复；
+ 若只有 fp16 异常，优先做 bf16/fp32 replay；
+ 若禁用 fused kernel 后稳定，优先查 kernel 或 dtype；
+ 若参数已污染，必须回滚到 first non-finite 之前的 checkpoint。

#### 7.9.8 某一层、某几个 head 或某几个 expert 异常
| 异常位置 | 重点检查 | 最小实验 | 修复方向 |
| --- | --- | --- | --- |
| 单层异常 | input/output、param group、grad/update | 冻结该层或降低该层 LR replay | 修初始化、LR group、局部实现 |
| 多层深度趋势异常 | residual RMS depth trend | 改 warmup/LR/residual scale 对照 | residual scaling、Pre-LN、DeepNorm 类方法 |
| 单 head attention 异常 | Q/K norm、score、entropy、mask | head-wise score logging | QK Norm、mask、RoPE |
| 多 head entropy collapse | score scale、temperature、long context | 调 temperature/QK scale replay | attention scale 修复 |
| 单 expert 异常 | expert load、expert grad、expert state | 固定 routing 或降低 expert LR | expert state、capacity、router |
| 多 expert collapse | router entropy、aux loss、drop token | 调 aux/z-loss/capacity | router 稳定性修复 |


判断一个局部异常是不是根因，需要看：

+ 该模块输入是否正常；
+ 该模块输出是否第一个异常；
+ 去掉或替换该模块后异常是否消失；
+ 该模块的 update-to-weight ratio 是否异常；
+ 异常是否与特定 batch、rank、position 或 expert load 相关。

#### 7.9.9 某些 batch 稳定触发异常
| 排查项 | 判断标准 | 操作 |
| --- | --- | --- |
| 是否同 batch 必炸 | replay 稳定复现 | 进入 batch 二分 |
| 是否少数样本触发 | top-loss 样本集中 | 去掉样本 replay |
| 是否极长样本触发 | 长度分桶后长样本异常 | length curriculum 或修长上下文 |
| 是否 packed 触发 | unpacked 不炸 | 修 packed boundary mask |
| 是否某 source 触发 | source ablation 后恢复 | 修数据源或采样权重 |
| 是否某 token 触发 | special token / pad token loss 异常 | 修 tokenizer/template/label |
| 是否某 rank batch 触发 | rank-local replay 复现 | 修 sampler 或 preprocessing |


临时处理可以使用 bad batch quarantine，但必须满足：

+ 记录被隔离样本的比例；
+ 隔离样本必须可追溯；
+ 隔离后不能让数据分布严重偏移；
+ 隔离只是临时措施，后续要修数据、mask 或 template 根因。

### 7.10 第八步：按根因类别设计最小验证实验
最小验证实验应尽量只改变一个因素。下面给出常见类别的验证方式。

| 疑似类别 | 最小验证实验 | 根因成立的判据 | 根因修复方向 |
| --- | --- | --- | --- |
| 数据问题 | bad batch replay；去掉 top-loss 样本；source ablation | 去掉异常样本或 source 后 spike 消失 | 清洗数据、修 label、修 weight、调整数据混合 |
| mask / packing | packed vs unpacked；toy packed batch；非法 attention 检查 | packed 炸而 unpacked 不炸，或 toy case 不通过 | 修 causal/padding/packed boundary mask、position id |
| label / loss | tiny batch overfit；hand-check CE；检查 ignore_index | tiny batch 不能 overfit或手算不一致 | 修 label shift、loss reduction、ignore_index |
| LR / warmup | LR 降 2–10 倍 replay；延长 warmup 对照 | 降 LR 后 update 和 loss 稳定 | 调 LR、warmup、schedule、batch-LR scale |
| optimizer state | reset state vs loaded state；检查 Adam m/v | reset 后稳定或 loaded state 异常 | 修 checkpoint、m/v、epsilon、beta、param group |
| param group | 打印每组 LR/WD；单独降低异常组 LR | 某组 U/W 异常且调组后恢复 | 修 LR group、weight decay、norm/bias decay 策略 |
| 初始化 / 尺度 | step0 forward/backward；检查 layer RMS | 初始层级 RMS 已异常 | 修初始化、residual scale、embedding/lm_head scale |
| 架构 | Pre-LN/QK Norm/residual scaling ablation | 特定架构改动稳定异常指标 | 采用结构性稳定设计 |
| 精度 | bf16/fp32 replay；关闭 autocast/fused kernel | 高精度稳定、低精度不稳定 | loss scaling、bf16、stable kernel、FP32 accumulation |
| 分布式 | 单卡 vs 多卡；all-reduce 前后对比；param checksum | 单卡稳定或 all-reduce 后异常 | 修 DDP/FSDP/ZeRO、通信 dtype、accumulation |
| MoE | 固定 routing；dense fallback；调 capacity / aux / z-loss | routing 改动显著影响异常 | 修 router、capacity、aux/z-loss、expert LR |
| 代码 bug | toy example、reference implementation 对比 | toy case 不符合数学预期 | 修实现，增加单元测试 |


最小验证实验的结果应被分成三类：

| 结果 | 含义 | 下一步 |
| --- | --- | --- |
| 明确消除异常 | 干预对象很可能接近根因 | 做正式修复和回归验证 |
| 只降低异常幅度 | 可能是症状缓解或部分根因 | 继续向上游查 first abnormal |
| 无影响 | 该假设优先级下降 | 切换诊断分支 |

### 7.11 第九步：找到根因后的修复原则
根因修复应优先处理产生异常的机制，而不是直接压制异常数值。

| 根因类别 | 修复优先级 |
| --- | --- |
| 数据 / label / sample weight | 清洗、修正、降权、重采样、template 修复 |
| mask / packing / position | 修单元测试，构造 toy batch 验证，再恢复训练 |
| LR / schedule | 修 global step、warmup、param group、batch-LR scale |
| optimizer | 修 state loading、epsilon、beta、AdamW 配置、state dtype |
| 初始化 / 架构 | 调 residual scale、QK scale、Pre-LN、DeepNorm 类稳定结构 |
| precision | 切 bf16、FP32 accumulation、stable softmax、修 fused kernel |
| distributed | 修 rank consistency、all-reduce、FSDP/ZeRO state、checkpoint |
| MoE | 修 router、capacity、load balancing、z-loss、expert state |
| 代码 bug | 增加 toy test、reference test、tiny overfit test |


修复后需要回归验证：

1. 异常 batch replay 不再触发；
2. 正常 batch 的 loss / grad / update 不恶化；
3. 指标分布回到稳定窗口；
4. valid loss 不异常上升；
5. clip rate、overflow rate、skip step 显著下降；
6. rank-wise 指标一致；
7. checkpoint resume 后曲线连续；
8. MoE 场景下 expert load、router entropy、dropped tokens 正常。

如果某个修复让训练“不炸了”，但 valid loss 明显变差、clip rate 仍然很高、attention entropy 长期异常或 expert specialization 消失，则说明修复可能只是新的数值压制，不是完整根因治理。

### 7.12 第十步：找不到根因时如何选择止血手段
如果根因暂时无法定位，或根因短期无法彻底解决，可以使用 B 类数值约束作为保险丝。但选择止血手段应遵循三个原则：

1. **离异常源头越近越好。**  
attention score 异常优先处理 Q/K 或 score，而不是只 clip global grad。  
optimizer update 异常优先处理 update 或 U/W，而不是只看 grad norm。
2. **扰动越小越好。**  
soft-capping 通常比 hard clipping 平滑；global grad norm clipping 通常比 per-value clipping 少破坏方向。
3. **必须记录触发率和副作用。**  
任何 clip / clamp / rescale 如果长期高频触发，都应被视为训练机制的一部分，而不是临时保险丝。

| 异常对象 | 优先止血手段 | 不建议优先使用 | 原因 |
| --- | --- | --- | --- |
| global grad norm | global grad norm clipping | per-value grad clipping | global clipping 对方向破坏较小 |
| 局部层 grad | layer-wise clipping | 全模型大幅降 LR | 局部问题应局部约束 |
| optimizer update | update clipping、U/W cap | 只 clip gradient | update 异常可能来自 Adam state，而非 grad |
| attention score | QK Norm、temperature、softcap | hard clamp score | hard clamp 可能破坏 attention 几何 |
| logits | logit soft-capping | hard logits clipping | softcap 对概率分布更平滑 |
| activation outlier | residual/MLP scaling、局部 activation clamp | 全局 hidden hard clamp | 全局 clamp 容易掩盖上游问题 |
| FP16 overflow | loss scaling、bf16、FP32 softmax | 忽略 overflow 继续 step | overflow 会污染参数和 state |
| bad batch | quarantine + 数据记录 | 静默丢弃 batch | 静默丢弃无法修根因 |
| MoE expert 爆 | expert clip、router z-loss、capacity 调整 | 只加大 global grad clip | MoE 问题通常是 routing 局部问题 |
| reward / RLHF outlier | reward normalization / clipping | 不查 reward model 直接 clip | reward clip 会改变优化目标 |


止血手段是否掩盖真实问题，可以用以下指标判断。

| 指标 | 危险信号 | 解释 |
| --- | --- | --- |
| clip rate | 长期高频触发 | clip 正在定义训练动力学 |
| clip 前后 direction cosine | cosine 很低 | 梯度方向被明显改写 |
| pre-clip norm trend | clip 后 loss 稳，但 pre-clip norm 继续上升 | 上游根因仍恶化 |
| activation/logits upstream | 下游被压住，上游 RMS 继续漂移 | clamp 只压症状 |
| valid loss | train loss 稳但 valid 恶化 | 数值稳定但学习质量下降 |
| entropy / calibration | logits 被压后 entropy 异常 | 输出分布被扭曲 |
| MoE expert load | 数值稳定但 expert 负载退化 | router 被过度约束 |
| overflow / skip | skip 减少但 update 异常仍在 | AMP 问题可能未根治 |
| bad batch ratio | quarantine 比例升高 | 数据管线问题未修 |

### 7.13 典型诊断案例
#### 案例 1：resume 后 loss 立即 spike
**现象**：训练从 checkpoint 恢复后，前一个 checkpoint 前 loss 平稳，resume 后 1–3 step 内 loss spike，grad clip 高频触发。

**排查顺序**：

1. 对比 resume 前后 global step；
2. 检查 scheduler step 和当前 LR；
3. 检查各 param group LR；
4. 检查 Adam m/v 是否正确加载；
5. 检查 loss scaler 和 overflow state；
6. 比较 resume 前连续训练的一步 update 与 resume 后第一步 update；
7. 若 update-to-weight ratio 明显变大，优先修 optimizer / scheduler state。

**错误止血方式**：直接把 grad clip 阈值调小。  
**正确修复方向**：恢复正确 scheduler、optimizer state、scaler state；必要时回滚到未污染 checkpoint。

---

#### 案例 2：某些 packed batch 稳定触发 spike
**现象**：loss spike 只在部分 batch 出现，replay 可复现；per-token loss 最高位置集中在 packed sequence 边界附近。

**排查顺序**：

1. 保存 batch 的 packed segment 边界；
2. 检查 causal mask 是否阻止跨样本 attention；
3. 检查 position id 是否在 segment 内 reset 或符合 recipe；
4. 比较 packed 与 unpacked replay；
5. 构造 toy packed batch，验证 token 不能 attend 到其他样本；
6. 修 mask 后 replay 原 bad batch。

**错误止血方式**：把这些 batch 直接丢弃。  
**正确修复方向**：修 packed boundary mask、position id 和对应单元测试。

---

#### 案例 3：grad norm 爆炸但 loss 没有先 spike
**现象**：loss 数值没有明显极端，但 backward 后 global grad norm 突然变大，部分 rank 更明显。

**排查顺序**：

1. 检查 loss reduction 是否按 token 数归一，而不是按 microbatch 数或 rank 数错误归一；
2. 检查 sample weight 是否极端；
3. 检查 ignore_index 是否失效；
4. 查看 layer-wise grad，是全层放大还是局部放大；
5. 查看 all-reduce 前后 grad norm；
6. 用 toy batch 手算 loss 和 gradient scale；
7. 单卡与多卡等价 batch 对比。

**错误止血方式**：只加大 global grad clipping。  
**正确修复方向**：修 loss normalization、sample weight、rank reduction 或 accumulation scaling。

---

#### 案例 4：attention entropy collapse 后 logits 极端
**现象**：某些层某些 head 的 attention entropy 持续下降，随后 logits max 增大，loss spike。

**排查顺序**：

1. 记录 head-wise entropy，而不是只看 layer mean；
2. 查看 Q norm、K norm 和 attention score p99；
3. 检查 RoPE / position id，尤其是长序列；
4. 检查 mask 是否异常；
5. 按 sequence length 分桶对比；
6. 尝试 QK Norm、attention temperature 或 score softcap 作为对照；
7. 验证 entropy、logits、valid loss 是否同时恢复。

**错误止血方式**：只 hard clip logits。  
**正确修复方向**：修 QK score 生成机制、position/mask 或 attention scale。

---

#### 案例 5：MoE expert collapse
**现象**：loss 波动增大，部分 expert token load 接近 0，少数 expert 负载过高且 grad norm 异常。

**排查顺序**：

1. 查看 router logits max/RMS；
2. 查看 router entropy；
3. 查看 expert load CV、dropped tokens、capacity usage；
4. 查看 aux load balancing loss 和 z-loss 是否过强或过弱；
5. 固定 routing replay；
6. 调 capacity factor、router temperature、z-loss 权重；
7. 查看 expert update-to-weight ratio。

**错误止血方式**：只对全部参数做 global grad clipping。  
**正确修复方向**：修 routing、capacity、aux/z-loss、expert LR 或 expert optimizer state。

---

### 7.14 诊断结束的接受标准
一次稳定性修复不能以“loss 暂时不炸”为唯一标准。建议使用以下接受标准。

| 检查项 | 接受标准 |
| --- | --- |
| 异常 replay | 原 bad batch 在同 checkpoint 下不再触发异常 |
| 正常 replay | 正常 batch 的 loss / grad / update 不恶化 |
| 指标分布 | residual RMS、attention score、logits、grad、update 回到稳定窗口 |
| optimizer | Adam m/v finite，step count 连续，U/W 合理 |
| precision | overflow / skip step 不再持续发生 |
| distributed | rank-wise loss/grad/checksum 一致 |
| MoE | expert load、router entropy、dropped tokens 正常 |
| stopgap | clip rate、softcap saturation、quarantine ratio 不高频 |
| 质量 | valid loss、held-out loss 或核心 eval 不异常下降 |
| resume | checkpoint round-trip 后曲线连续 |


如果修复后仍然满足以下任一情况，应继续诊断：

+ clip rate 长期处于高频；
+ pre-clip norm 继续上升；
+ logits / attention entropy / residual RMS 仍然漂移；
+ valid loss 变差；
+ 某 rank 或某 expert 持续异常；
+ 修复只在某个 batch 生效，但新 batch 继续触发同类问题；
+ 需要越来越强的 clamp / clip 才能维持训练。

---

### 7.15 本章小结
大模型训练稳定性的诊断流程可以概括为：

```plain
先保存现场，
再判断异常发生阶段，
再定位异常空间位置，
再做最小验证实验，
再修复根因，
最后才选择最小扰动的止血机制。
```

其中最关键的工程判断是：

+ **loss spike 不等于数据问题**，也可能是 optimizer update、precision、distributed 或 MoE routing 问题；
+ **grad norm 爆炸不等于梯度本身是根因**，它可能是 activation、loss、mask 或 update 的下游症状；
+ **clip 能救训练，但不能证明问题已解决**；
+ **update-to-weight ratio 往往比 grad norm 更接近真实优化风险**；
+ **rank-wise、layer-wise、head-wise、expert-wise 的局部异常经常被 global mean 掩盖**；
+ **所有稳定性修复都必须通过 replay、ablation 和回归指标验证**。

因此，一个成熟的训练稳定性系统，不只是监控 loss 和 grad norm，而是能够把一次异常分解为：**哪个 batch、哪个 rank、哪个阶段、哪个层/头/expert、哪个对象、哪个状态转移首先偏离正常范围**。只有定位到这个“第一个异常状态转移”，才能区分真正的根因修复与单纯的数值压制。
